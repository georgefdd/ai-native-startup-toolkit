# 各阶段退出标准详表

## Idea → MVP 退出标准

必须三个问题都回答"是"：

1. **问题是否真实且具体？**
   - 能准确说出谁会遇到这个问题
   - 多久发生一次
   - 有多严重
   - 他们目前怎么应对

2. **你的方案是否解决了实际的问题？**
   - 不是你最初假设的问题
   - 而是验证过程揭示的问题
   - 有时这两者相同，但不是总是

3. **你有足够的信号来证明值得构建吗？**
   - 永远不会有确定性
   - 等待确定性本身就是一种失败模式
   - 需要足够的定性证据使MVP成为理性决策而非信仰之举

**退出产物**：
- 问题-方案契合声明
- 至少20段一手用户对话原文
- 轻量原型（用于用户对话的道具，不是产品）

---

## MVP → Launch 退出标准

**PMF证据**：真实用户群体觉得产品有价值到愿意回来用、付费、推荐

判断方法：
- Sean Ellis测试：>40%活跃用户答"非常失望"
- 努力曲线：产品开始自己拉用户回来，不再需要创始人不停推

**退出产物**：
- 留存数据曲线
- 付费转化证据
- 用户推荐证据（NPS或自发推荐）
- 最小化安全审查通过

---

## Launch → Scale 退出标准

三个要素必须同时满足：

1. **增长可重复且渠道驱动**
   - 不仅在留存用户，还在通过特定渠道可预测地获取新用户
   - CAC、LTV、回本周期是已知可防御的数字

2. **产品能承载生产负载**
   - 基础设施已加固
   - 安全和合规就绪
   - 可靠性在真实生产条件下成立

3. **运营不再依赖创始人瓶颈**
   - 流程存在，自动化就位
   - 你不再是亲自处理客服、分诊、冲刺计划或报告的人

**退出产物**：
- 可重复的增长渠道数据
- 生产级基础设施 + 安全审查
- 运营自动化系统（创始人不在也能运转）

---

## Scale 退出（公司成熟）

不是单一里程碑，而是阈值事件——公司在创始人越来越不直接运营日常的情况下也能持续：

三种典型形式：
1. 可持续盈利（不再需要外部资本）
2. IPO就绪
3. 被收购

**必须能回答**："如果一个资金雄厚的现有玩家今天复制了你的产品，你的用户会留下来吗？"
